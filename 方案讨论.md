# Agent Team 研发生产流程平台方案（V0.2）

## 1. 目标与边界

### 1.1 目标
- 建立一套可配置、可审计、可复用的研发流程平台。
- 覆盖四个阶段：需求分析、技术方案、开发、测试。
- 提供基于 Vue 的前端界面承载配置、执行与审核。
- 以 TDD 作为研发默认方法：技术方案完成后先设计测试用例，再进入开发。
- 每个阶段都具备：
  - 明确输入（Input）
  - 可追溯产出物（Deliverables）
  - 人工检查/签核（Human Checkpoint）
  - 可配置 Agent 与 Skill
- 执行形态统一为：**终端启动 Claude，多 Agent Team 协作，tmux 分屏展示控制面板**。

### 1.2 非目标（当前版本不做）
- 不做复杂 BPMN 级编排引擎（先支持平台内的可视化动态编辑 + YAML 双轨）。
- 不做跨组织权限系统（先支持单团队/单项目）。
- 不直接替代 CI/CD 平台（先与现有 CI 做最小集成）。

---

## 2. 核心概念模型

### 2.1 对象定义
- **Workflow**：一个完整研发流程（含多个 Stage）。
- **Stage**：流程阶段（需求分析/技术方案/开发/测试）。
- **Task**：阶段内可执行任务单元（可并行或串行）。
- **Agent Profile**：Agent 角色配置（提示词、模型、权限、工具白名单）。
- **Agent Responsibility**：可页面编辑的职责说明（目标、边界、不做事项、交接条件）。
- **Skill Pack**：可挂载能力集（规则、脚本、模板、检查器）。
- **Artifact**：产出物（文档、代码、测试报告、审查记录）。
- **Human Gate**：人工审核门（通过/驳回/补充）。

### 2.2 状态机（简化）
- `pending` -> `running` -> `await_human_check` -> `approved` -> `completed`
- 驳回路径：`await_human_check` -> `rework_required` -> `running`
- 失败路径：`running` -> `failed`

## 3. 四阶段输入与产出设计（TDD 驱动）

## 3.1 需求分析阶段（Stage: requirement_analysis）
- **输入**
  - 业务目标/用户痛点
  - 背景资料（竞品、现网数据、历史需求）
  - 约束条件（时间、人力、合规）
- **Agent Team 建议**
  - `PM-Agent`：需求拆解与优先级
  - `Domain-Agent`：业务语义校验
  - `Risk-Agent`：范围/依赖/冲突识别
- **关键 Skill**
  - 需求模板生成
  - 术语标准化
  - 风险清单提取
- **产出物**
  - `PRD.md`
  - `Scope.md`
  - `RiskRegister.md`
  - `需求评审记录.md`
- **人工检查点**
  - 角色：产品负责人 + 技术负责人
  - 检查项：目标清晰、范围边界、验收标准可测
  - 通过后进入技术方案；驳回则回流补齐。

## 3.2 技术方案阶段（Stage: technical_design）
- **输入**
  - `PRD.md`、`Scope.md`、`RiskRegister.md`
  - 当前系统架构与约束
- **Agent Team 建议**
  - `Architect-Agent`：总体架构与模块边界
  - `Security-Agent`：安全/权限/数据风险审查
  - `Cost-Agent`：性能与成本估算
- **关键 Skill**
  - ADR 模板生成
  - 架构对比分析
  - 威胁建模清单
- **产出物**
  - `Architecture.md`
  - `ADR/*.md`
  - `API-Spec.yaml`
  - `DataModel.md`
  - `TestCases.md`（面向需求验收点的测试用例设计，TDD 前置产物）
  - `TechReview.md`
- **人工检查点**
  - 角色：架构师 + 资深工程师
  - 检查项：可实施性、可扩展性、安全性、成本合理性、测试可实现性
  - 通过后进入开发阶段。

## 3.3 开发阶段（Stage: development）
- **输入**
  - 技术方案产出物（Architecture/ADR/API）
  - `TestCases.md`（必须）
  - 开发任务拆解（Issue/任务清单）
- **Agent Team 建议**
  - `Code-Agent`：编码实现
  - `Review-Agent`：静态审查与风格一致性
  - `Refactor-Agent`：复杂度治理与重构建议
- **关键 Skill**
  - 代码脚手架生成
  - 测试先行模板（red-green-refactor）
  - 规范检查（lint/type）
  - 提交信息与变更摘要生成
- **产出物**
  - 代码变更（PR）
  - 单元测试与必要集成测试（先有失败测试，再实现通过）
  - `变更点审查.md` 更新
  - `DevNotes.md`（关键设计取舍）
- **人工检查点**
  - 角色：Reviewer（至少 1 人）
  - 检查项：是否遵循 TDD、功能正确性、测试充分性、回归风险
  - 通过后进入测试阶段。

## 3.4 测试阶段（Stage: testing）
- **输入**
  - 已通过审查的代码分支
  - `TestCases.md`
  - 测试计划/验收标准
- **Agent Team 建议**
  - `TestDesign-Agent`：测试用例设计
  - `TestExec-Agent`：自动化执行与结果汇总
  - `Quality-Agent`：缺陷归因与风险评级
- **关键 Skill**
  - 用例生成器
  - 覆盖率/失败聚类分析
  - 回归清单管理
- **产出物**
  - `TestPlan.md`
  - `TestReport.md`
  - 覆盖率报告
  - 缺陷清单（含优先级）
- **人工检查点**
  - 角色：QA + 研发负责人
  - 检查项：关键路径通过、阻塞缺陷清零、发布建议明确
  - 通过后可进入发布流转。

## 4. 前端承载（Vue）

### 4.1 前端定位
- 前端作为流程控制台，承载配置、执行监控、人审与审计查询。
- 后端编排仍以终端 Claude + tmux 运行为核心，前端通过 API 发起与查看状态。

### 4.2 前端核心页面
- `流水线管理`：创建/复制/启停流水线，支持动态增删阶段与顺序调整。
- `阶段编排`：为每个阶段动态添加/移除 Agent，配置并发策略。
- `Agent 职责编辑`：在线编辑职责、输入期望、交付标准、失败回退策略。
- `Skill 平台管理`：Skill 注册、版本管理、启停、权限范围（项目/组织）管理。
- `执行看板`：查看 tmux 会话映射、阶段状态、产出物清单、人审门状态。
- `审计中心`：按流程/阶段/Agent 检索输入、产物、审批、日志。

### 4.3 技术栈建议
- `Vue 3` + `TypeScript` + `Pinia` + `Vue Router`
- UI：`Element Plus`（MVP 快速搭建）
- 通信：REST + SSE（状态推送）或 WebSocket（后续）

## 5. 执行与编排：终端 Claude + Agent Team + tmux

### 4.1 运行形态
- 每个 Stage 都由终端命令触发，例如：
  - `agentflow run --stage requirement_analysis`
  - `agentflow run --stage technical_design`
  - `agentflow run --stage development`
  - `agentflow run --stage testing`
- 命令内部启动 Claude 会话，并按 Stage 配置组建 Agent Team。

### 4.2 tmux 控制面板布局（建议）
- Pane 1（左上）：主控制台（Orchestrator 日志）
- Pane 2（右上）：当前 Stage 的主 Agent 输出
- Pane 3（左下）：审查门状态（待审/通过/驳回）
- Pane 4（右下）：Artifacts 索引与实时生成列表

### 4.3 协作机制
- Orchestrator 根据配置分配任务给多个 Agent。
- Agent 以共享上下文仓（文件 + 中间摘要）协作。
- 必要时触发“讨论回合”（例如方案冲突）并形成决议记录。

## 6. 配置设计（草案）

```yaml
workflow:
  id: default-rnd-flow
  stages:
    - id: requirement_analysis
      agents: [pm_agent, domain_agent, risk_agent]
      skills: [prd_template, term_normalizer, risk_extractor]
      inputs: [brief.md, constraints.md]
      outputs: [PRD.md, Scope.md, RiskRegister.md, review/req-review.md]
      human_gate:
        required: true
        reviewers: [product_owner, tech_lead]
    - id: technical_design
      agents: [architect_agent, security_agent, cost_agent]
      skills: [adr_generator, arch_compare, threat_model]
      inputs: [PRD.md, Scope.md, RiskRegister.md]
      outputs: [Architecture.md, ADR/, API-Spec.yaml, DataModel.md, TestCases.md, TechReview.md]
      human_gate:
        required: true
        reviewers: [architect, senior_engineer]
    - id: development
      agents: [code_agent, review_agent, refactor_agent]
      skills: [tdd_template, lint_checker, commit_summarizer]
      inputs: [Architecture.md, ADR/, API-Spec.yaml, TestCases.md]
      outputs: [src/, tests/, DevNotes.md]
      tdd:
        required: true
        red_green_refactor: true
  ui:
    editable_pipeline: true
    editable_agent_responsibility: true
  skill_registry:
    enabled: true
    scopes: [project, org]
```

## 7. 人工审核机制（必须项）

- 每个阶段必须 `human_gate.required = true`。
- 未通过前，禁止进入下阶段（硬门禁）。
- 审核结果结构化记录：
  - `decision`: approve/reject
  - `comments`: 审核意见
  - `required_fixes`: 必改项列表
  - `signed_by`: 审核人
  - `timestamp`

---

## 8. TDD 执行规范（新增）

- 技术方案阶段必须交付 `TestCases.md`，作为开发阶段准入条件。
- 开发阶段默认执行 `Red -> Green -> Refactor` 循环。
- 代码提交时校验：
  - 是否存在与需求映射的测试用例
  - 是否存在先失败后通过的测试轨迹（日志或提交序列）
  - 覆盖率是否达到门槛（阈值可配置）
- 人审关注点：
  - 测试是否先于实现定义行为
  - 变更是否由测试驱动而非仅补测

---

## 9. 可追溯性与审计

- 全链路记录：
  - 输入版本（来源、时间、提交号）
  - Agent 执行日志
  - Skill 调用记录
  - 产出物哈希与版本
  - 人审结论与签核人
- 支持问题追溯：
  - “某个需求为何这样实现？”
  - “哪位 Agent 生成了此方案？”
  - “哪次人审允许该风险通过？”

## 10. 最小可行版本（MVP）建议

### 8.1 第一阶段（2~3 周）
- 支持 4 个默认 Stage + 基础动态编辑（增删 Agent、编辑职责）。
- 前端 Vue 控制台上线（流水线管理、Agent 职责编辑、Skill 管理、执行看板）。
- 配置文件驱动（YAML）与页面配置双向同步（以平台配置为准）。
- tmux 面板 + 终端命令行执行。
- 每阶段产出文档 + 人审门禁跑通。
- TDD 约束生效：技术方案后先产测试用例再开发。

### 8.2 第二阶段（3~6 周）
- 引入并行任务与失败重试策略。
- 接入 Git/PR 与测试平台结果回写。
- 增加指标看板（周期、返工率、缺陷密度）。

## 13. 外部协作文档同步约定（新增）

- 当前方案讨论主文档以仓库内 `方案讨论.md` 为准。
- 对外同步到飞书等平台时，采用“仓库文档生成 -> 人工确认 -> 外部发布”的流程。
- 若自动化浏览器写入能力不可用，平台仍可提供“一键复制块”（Markdown/富文本）以便人工粘贴发布。

## 14. Plan 驱动开发与项目空间（新增）

### 14.1 开发推进方式
- 虽然执行端由终端 Agent 驱动，但工程推进采用“计划先行”的节奏：
  1. 先明确阶段目标、输入、验收标准（Plan）
  2. 再按 TDD 先写测试用例与失败测试（Red）
  3. 最后最小实现并通过测试（Green + Refactor）
- 每轮迭代都必须更新文档与审查记录，保证跨会话可追溯。

### 14.2 项目空间目录（当前）
- `frontend/`：Vue 前端控制台（流水线/Agent/Skill/执行看板）
- `backend/`：API 服务与鉴权、配置存储
- `orchestrator/`：终端执行编排（Claude Agent Team、tmux 协调）
- `skills/`：Skill 模板、版本和元数据
- `configs/`：流水线配置、阶段模板、Agent 职责模板
- `tests/`：跨模块测试与集成测试
- `docs/`：设计文档、迭代记录、发布说明
- `scripts/`：本地开发与 CI 脚本

### 14.3 GitHub 交付约定（你后续上传用）
- 建议主分支：`main`
- 分支策略：`feature/*`、`fix/*`、`chore/*`
- PR 合并门禁：
  - 单元测试通过
  - 关键路径测试通过
  - 审查记录齐全（包含 `变更点审查.md` 更新）
- 首次发布建议包含：
  - `README.md`
  - 本文档 `方案讨论.md`
  - `变更点审查.md`
  - 基础工程目录与初始化脚本
